Ingress 是什么？

发表于2026-03-18|更新于2026-04-09|k8s
|浏览量:

ingress

实现：流量控制、负载均衡

官网：https://kubernetes.io/zh-cn/docs/concepts/services-networking/ingress/

  • 使用一种能感知协议配置的机制来解析 URI、主机名称、路径等 Web 概念， 让你的 HTTP（或 HTTPS）网络服务可被访问。 Ingress 概念允许你通过 Kubernetes API 定义的规则将流量映射到不同后端。
  • Ingress 是对集群中服务的外部访问进行管理的 API 对象，典型的访问方式是 HTTP。
  • Ingress 可以提供负载均衡、SSL 终结和基于名称的虚拟托管。

Ingress 是什么？

Ingress 提供从集群外部到集群内服务的 HTTP 和 HTTPS 路由。 流量路由由 Ingress 资源所定义的规则来控制。

    简单来说，Ingress 是 K8s 集群中管理“外部访问”进入“内部服务”的一层路由规则。

    1. 为什么要用 Ingress？

    在没有 Ingress 之前，我们通常用 NodePort 或 LoadBalancer 暴露服务：

    • NodePort：每个服务都要开一个端口（如 30001, 30002），端口多了根本记不住，而且不安全。
    • LoadBalancer：在云厂商那里，每创建一个服务都要申请一个独立的公网 IP，这太贵了！

    Ingress 的出现解决了这些痛点：

    • 单一入口：只需要一个公网 IP 或一个入口点，就能转发给集群内的几十个服务。
    • 域名分发：可以根据域名（host）和路径（path）来分发流量。
    • 安全管理：集中管理 SSL/TLS 证书（HTTPS）。

2. Ingress 的核心组成

理解 Ingress 需要区分两个概念：

组件 角色 比喻
Ingress 资源 配置文件（YAML） 告诉保安：“如果有人找 A 单元，领他去 1 号楼”。
Ingress Controller 实际干活的软件（如 Nginx） 站在门口负责看管和领路的“保安大叔”。

注意： K8s 默认只定义了 Ingress 的标准，不提供控制器。你必须自己安装一个控制器（最常用的是 Ingress-Nginx）才能让规则生效。

3. 它能实现什么功能？

路由分发（基于域名和路径）

你可以配置如下规则：

  • example.com/api -> 转发到 api-service
  • example.com/web -> 转发到 web-service
  • shop.com -> 转发到 shopping-service

HTTPS 卸载

你不需要在每个微服务里配置证书。只需在 Ingress 层配置一次 TLS 证书，外部访问是加密的，内部流量可以走轻量级的 HTTP。

灰度发布 / 负载均衡

高级的 Ingress Controller（如 Traefik 或 Kong）还可以支持蓝绿发布、金丝雀发布或限流。

5. 总结：Ingress 的工作流程

  1. 用户在浏览器输入域名。或者基于url（全球统一资源定位符）的流量引导

  2. DNS 解析到 Ingress Controller 的公网 IP。

  3. Ingress Controller 接收请求，查看自己的“导视牌”（Ingress 规则）。

  4. 匹配成功后，直接将请求转发到集群内部对应的 Service 或 Pod。

1. 基于域名的虚拟主机路由 (Name-based Virtual Hosting)

这种模式下，你拥有一个公网 IP，但绑定了多个域名。Ingress 根据用户访问的是哪个 域名 来决定去哪个服务。

  • 场景：你在同一个集群里运行了“官网”和“博客”。
  • 流程：
    1. 用户访问 app.com -> 路由到 Service A。
    2. 用户访问 api.com -> 路由到 Service B。
  • YAML 关键点：在 rules 下定义多个不同的 host。

2. 基于 URL 路径的路由 (Path-based Routing)

这种模式下，域名只有一个，但根据 URL 后缀（路径）的不同，请求会被分流到不同的微服务。

(https://domain.com/video) and domain.com/images being routed to separate backend services]

  • 场景：单页应用（SPA）。前端静态资源在一个服务，后端 API 在另一个服务。
  • 流程：
    1. 用户访问 example.com/web -> 路由到 Frontend Service。
    2. 用户访问 example.com/api -> 路由到 Backend Service。
  • YAML 关键点：在同一个 host 下定义多个 paths，并设置 pathType: Prefix（前缀匹配）。

两种模式对比表

特性 基于域名 (Host-based) 基于路径 (Path-based)
入口域名 多个域名 (a.com, b.com) 单个域名 (example.com)
IP 消耗 共享 1 个公网 IP 共享 1 个公网 IP
SSL 证书 通常需要泛域名证书或多个证书 只需要 1 个证书
适用场景 多租户、完全独立的不同业务 微服务架构、前后端分离

混合模式（进阶版）

在实际生产中，我们经常混合使用。例如：

  • dev.company.com/api -> 转发到测试版后端
  • prod.company.com/api -> 转发到正式版后端
