🚨 Kubernetes 生产级故障复盘：etcd 损坏与集群“脑死亡”修复

发表于2026-03-18|更新于2026-04-09|故障排查
|浏览量:

  1. 这次由于 XTP/XFTP 导致电脑意外重启，随后引发的 Kubernetes 集群“崩塌与重生”是一次非常经典的运维故障恢复案例。

    底层原因是电脑重启导致 Master 节点的 etcd 数据损坏，进而引发了从系统镜像到网络插件，再到核心服务的连锁反应。以下是详细的事件复盘与解决步骤总结：

    🛑 事件背景：从重启到“脑死亡”

    • 起因：XFTP 导致系统崩溃重启，Master 节点上的 etcd 数据库文件损坏。
    • 症状：kubectl 命令失效（无法连接 API Server），控制面 Pod（apiserver/controller-manager）陷入死循环重启或直接无法启动。

    🛠️ 详细解决步骤与预期输出

    第一阶段：抢救“大脑”（etcd 与控制面）

    由于没有备份，我们采取了最彻底的“重置”方案。

    1. 操作：清理损坏的 etcd 数据目录 rm -rf /var/lib/etcd。
    2. 现象：API Server 恢复，但集群变成了“白纸”，所有之前的配置（Deployment, Service, RBAC）全部消失。
    3. 修复管理权限：由于认证信息丢失，重新绑定了 ClusterRole。
      • 预期输出：kubectl get nodes 能看到节点，但状态通常为 NotReady。

    第二阶段：修复“神经系统”（网络插件 Calico）

    etcd 重置后，Calico 失去了之前的网段记录，且默认配置与你的集群物理网段冲突。

    1. 操作：修改 calico.yaml，取消 CALICO_IPV4POOL_CIDR 的注释，并改为 10.244.0.0/16。
    2. 关键逻辑：让 Calico 的虚拟网段与 kubeadm init 时定义的 pod-network-cidr 保持一致。
    3. 预期输出：执行 kubectl apply -f calico.yaml 后，calico-node 开始尝试启动。

    第三阶段：疏通“血管”（Kube-proxy 与镜像源）

    由于网络限制，集群无法从海外官方仓库拉取 v1.35.2 镜像。

    1. 操作：修改 kube-proxy 的 DaemonSet，将镜像地址从 registry.k8s.io 改为阿里云镜像 registry.aliyuncs.com/google_containers，并将版本对齐为本地已有的 v1.35.0。
    2. 预期输出：所有节点上的 kube-proxy 瞬间变为 1/1 Running。

    第四阶段：重置“导航仪”（CoreDNS）

    etcd 清空后，集群内网域名解析服务彻底消失。

    1. 操作：
      • 执行 kubeadm init phase addon coredns 强制重建。
      • 再次修改 CoreDNS 的 Deployment 镜像地址为阿里云源，对齐版本 v1.13.1。
    2. 预期输出：kubectl get pods -n kube-system 中出现两个 coredns Pod，且状态为 1/1 Running。

    第五阶段：恢复“秩序”（Master 污点与 Node 状态）

    1. 操作：重新为 Master 打上 NoSchedule 污点，防止业务 Pod 占用核心资源。
    2. 预期输出：kubectl describe node k8s-master-1 看到 Taints 存在；所有节点状态变为 Ready。

    📈 最终健康状态清单

    检查项 验证命令 预期标准输出
    节点状态 kubectl get nodes 所有节点状态均为 Ready
    基础组件 kubectl get pods -n kube-system 所有 Pod（calico, proxy, coredns, etcd, api）均为 1/1 Running
    网络网段 kubectl get nodes -o wide 每个节点都有正确的 Internal-IP 且 PodCIDR 为 10.244.x.0/24
    DNS 解析 nslookup kubernetes.default 能够返回 Service IP（通常是 10.96.0.1）

    💡 经验教训与避坑指南

    1. ** etcd 必须备份**：这次是因为没有备份才走这么复杂的重建路线。建议定期执行：

      etcdctl snapshot save backup.db。

    2. 镜像源持久化：国内环境安装 K8s 时，尽量在 kubeadm-config 中固定使用阿里云镜像源，避免组件重启后私自去 registry.k8s.io 拉取。

    3. YAML 备份：所有的核心组件 YAML（Calico, Ingress, 业务应用）都应保存在 Git 或本地目录， etcd 炸了后可以快速 kubectl apply 恢复业务。

1. etcd：集群的“心脏”

崩溃原因

  • 物理损坏：电脑意外重启导致 etcd 正在写入的 WAL（预写日志）文件断电损坏。
  • 数据不一致：重启后旧的数据库文件无法通过校验，导致 etcd 静态 Pod 无法启动。

排查方法

  • 查看系统日志：journalctl -u etcd 或查看 /var/log/pods/ 下的 etcd 日志。
  • 检查文件：ls -lh /var/lib/etcd 观察文件大小是否异常。

方案选择

  • 方案 A（恢复）：如果有备份，执行 etcdctl snapshot restore。（最优，但你当时没有）。
  • 方案 B（重置）：删除数据目录强制重建。（最快，但会导致所有配置丢失）。

2. kube-apiserver：集群的“大脑”

崩溃原因

  • 依赖失效：apiserver 必须连接 etcd 才能工作。etcd 不启动，apiserver 就会无限重启。
  • 认证失效：etcd 重置后，原本的 ServiceAccount 令牌失效，导致组件间无法握手。

排查方法

  • 静态 Pod 检查：crictl ps -a | grep apiserver。
  • 端口监听：netstat -lntp | grep 6443。

修复动作

  • 确保 etcd 运行正常后，apiserver 会自动恢复。
  • 若权限报错，需重新创建 ClusterRoleBinding（如你之前做的 admin-binding）。

3. Calico (CNI)：集群的“神经系统”

崩溃原因

  • 网段冲突：etcd 丢了，Calico 默认尝试使用 192.168.0.0/16，而你物理节点预留的是 10.244.0.0/24。
  • 初始化失败：calico-node 启动时会检查 etcd 里的 ClusterInformation，找不到就会报 connection refused 或 Init:Error。

排查方法

  • kubectl describe pod -n kube-system <calico-pod>。
  • kubectl logs -n kube-system <calico-pod> -c install-cni。

方案选择

  • 手动指定 IPPool：修改 YAML 里的 CALICO_IPV4POOL_CIDR。这是解决“网段漂移”的唯一办法。

4. kube-proxy：集群的“血管”

崩溃原因

  • 镜像版本漂移：etcd 消失后，由于没有配置文件约束，系统尝试拉取默认的最新版（v1.35.2）。
  • 网络隔离：默认去 registry.k8s.io 拉取，国内环境无法直接访问，导致 ImagePullBackOff。

排查方法

  • kubectl get ds -n kube-system 观察 AVAILABLE 数量。
  • kubectl get events -n kube-system 查看拉取失败记录。

方案选择

  • 修改 DaemonSet：将镜像地址硬编码为国内源（如阿里云）。这是最稳定的方案，因为 DaemonSet 会自动同步到所有节点。

5. CoreDNS：集群的“导航仪”

崩溃原因

  • 记录消失：CoreDNS 作为一个 Deployment 存储在 etcd 中，etcd 一空，它就彻底消失了。
  • 解析失败：即便 Pod 起来了，如果它找不到 API Server 的 IP，也会报错。

排查方法

  • kubectl get svc -A 发现没有 kube-dns 服务。
  • 在业务 Pod 内执行 nslookup 报错。

修复动作

  • 使用 kubeadm init phase addon coredns 重新注入配置。

🏁 总结：如何避免下一次灾难？

1. 备份策略 (3-2-1 原则)

  • etcd 定期快照：每天执行一次 etcdctl snapshot save。
  • YAML 离线化：所有的 install.yaml（Calico, Ingress 等）必须在本地磁盘有备份，不能只靠 kubectl get。

2. 镜像本地化

  • 在所有节点的 /etc/containerd/config.toml 中配置 镜像加速器。
  • 或者搭建私有镜像仓库（Harbor），将所有 K8s 核心镜像转存到私有仓库。

3. 运维排查优先级

当集群崩溃时，遵循此顺序：

  1. 存储层 (etcd) -> 2. 控制层 (apiserver) -> 3. 网络层 (CNI/Proxy) -> 4. 服务发现层 (DNS) -> 5. 业务层 (Ingress/Pods)。
