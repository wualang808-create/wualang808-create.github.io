  1. <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">containerd</font> 是 Docker 的底层容器运行时，负责实际执行容器的创建、启动、销毁等底层操作，<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">dockerd</font> 仅负责高层管理；
  2. <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">containerd.sock</font> 是 <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">dockerd</font> 和 <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">containerd</font> 之间的本地进程通信套接字，是二者的专属 “通信桥梁”；
  3. <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">dockerd</font> 启动参数 <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">--containerd=/run/containerd/containerd.sock</font>，本质是指定 dockerd 与 containerd 的通信接口，没有这个配置，dockerd 无法调用底层容器操作，Docker 服务完全无法工作。

这个架构设计也是云原生的核心思路 ——解耦，将高层管理和底层执行分离，让每个组件专注自身职责，既提升了 Docker 服务的稳定性，也让 <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">containerd</font> 可以独立被 Kubernetes 等其他云原生工具调用，成为行业通用的容器运行时。

  1. containerd
    全称：containerd Daemon（容器运行时守护进程）
    核心定位：容器运行时主进程，后台常驻运行（系统级守护进程），是 dockerd 与底层容器操作的核心中间层。
    核心职责：全局层面管理所有容器 / 镜像，包括镜像拉取 / 解压、容器创建 / 调度、容器资源全局管控、与 dockerd/k8s 通信，以及为每个容器创建并管理对应的 containerd-shim 进程。
    运行特征：单实例常驻（一台主机仅一个 containerd 主进程），不直接挂载容器命名空间、不成为容器进程的父进程。
  2. containerd-shim
    全称：containerd Shim Process（容器运行时垫片进程）
    核心定位：每个容器的专属子进程，containerd 的「代理 / 垫片」，为单个容器提供进程托管、日志重定向、资源隔离桥接等能力。
    核心职责：单个容器层面的本地管理，包括托管容器进程、保持容器命名空间存活、重定向容器日志 / 标准输出、收集容器进程状态、作为 runc 与 containerd 的通信桥梁。
    运行特征：一容器一 shim（一台主机有多少个运行中的容器，就有多少个 containerd-shim 进程），容器停止后对应的 shim 进程也会退出，轻量级无常驻。
  3. runc run container
